## Bugs found and fixed during the authenticated browser smoke test

1. **`/admin/calendar` crashed for every user (server runtime error).** `page.tsx` passed a plain `formatDateTime` function as a prop to the `CalendarEventRow` client component, which Next.js disallows across the Server/Client Component boundary. Fixed by moving the function to be defined locally inside the client component instead.
2. **Admin announcement creation failed for every user, including super admins.** `createAnnouncementAction` chained `.select('id').single()` after inserting into `announcements`, hitting the exact same Postgres RLS + `RETURNING` interaction already discovered and fixed for `calendar_events` in an earlier task (the SELECT policy's self-referential subquery can't see a row the same INSERT statement just wrote) — but the fix had never been applied here. Fixed the same way: generate the id client-side, never chain `.select()` after the insert. `learning-groups` and the rest of `announcements/actions.ts` were checked and found already safe.
3. **`PushSubscriptionControls.tsx` had a real React hydration mismatch on every page load**, found during the dedicated Web Push browser verification pass. `isAvailable` (browser feature + VAPID key detection) was computed via a synchronous `useMemo` calling `isPushSupported()` during render; that function reads `window`/`navigator`, so it returned `false` during SSR but `true` on the client's first render, mismatching the enable button's `disabled` attribute (visible as a repeatable React console error on every load). Fixed with `useSyncExternalStore` and an explicit `false` server snapshot — the standard pattern for values that legitimately differ between server and client. An initial `useEffect` + `setState` fix was rejected by the `react-hooks/set-state-in-effect` lint rule and replaced with this approach.
4. **`CalendarDateNavigator.tsx`'s date-range label rendered in reversed visual order under RTL**, found during the Admin Calendar Views v2 browser smoke test. The week/day/month label span had no explicit `dir`, so under the page's `dir="rtl"` context the browser's Unicode bidi algorithm displayed a range like `"5.7 - 11.7.2026"` (correct DOM text, confirmed via direct inspection) visually as `"1.7.2026 - 5.7"` (reversed). Fixed by adding `dir="ltr"` to that span, since date/time tokens are always LTR-formatted regardless of UI locale. Verified fixed live: the same week now renders in the correct visual order.
5. **The audit log's `toDate` filter silently excluded real same-day rows**, found during the Admin Audit Log Viewer v2 browser smoke test, in both `getAdminAuditLogs` (`admin-queries.ts`) and the CSV export route (`route.ts`). Both computed the exclusive next-day upper bound via `new Date(`${toDate}T00:00:00`)` — a string with no `Z`/offset, parsed by Node as **local server time**. On this server (`Asia/Jerusalem`, UTC+3), converting that local-time `Date` back to UTC via `.toISOString()` silently shifted the boundary earlier by the timezone offset, so filtering to "today" excluded rows created earlier that same day (confirmed live: `toDate=2026-07-09` showed 0 of 28 real rows, `toDate=2026-07-10` showed all 28). `fromDate` was unaffected, since it builds its UTC boundary directly as a string without any local-time `Date` parsing. Fixed in both files by computing the boundary entirely in UTC via `Date.UTC(year, month - 1, day + 1)`. Verified fixed live in both the page and a direct `fetch` against the export route.

All five fixes were verified live in the browser afterward. The latest Web Push v1 task added one migration and reran `supabase db reset`, type generation, `npm run check:no-hebrew-in-code`, `npm run lint`, `npm run build`, and `git diff --check`; all passed, with line-ending warnings only on `git diff --check`. The dedicated Web Push browser verification pass, the Admin Calendar Views v2 smoke test, the Learning Groups Timetable Views v2 smoke test, and the Admin Audit Log Viewer v2 smoke test each reran `npm run check:no-hebrew-in-code`, `npm run lint`, `npm run build`, and `git diff --check` after their fixes (or, for the learning-groups pass, with no fixes needed); all passed (no schema changes in any pass). The calendar pass also trimmed one pre-existing trailing-whitespace line in this file to keep `git diff --check` clean.

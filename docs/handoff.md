## Bugs found and fixed during the authenticated browser smoke test

1. **`/admin/calendar` crashed for every user (server runtime error).** `page.tsx` passed a plain `formatDateTime` function as a prop to the `CalendarEventRow` client component, which Next.js disallows across the Server/Client Component boundary. Fixed by moving the function to be defined locally inside the client component instead.
2. **Admin announcement creation failed for every user, including super admins.** `createAnnouncementAction` chained `.select('id').single()` after inserting into `announcements`, hitting the exact same Postgres RLS + `RETURNING` interaction already discovered and fixed for `calendar_events` in an earlier task (the SELECT policy's self-referential subquery can't see a row the same INSERT statement just wrote) — but the fix had never been applied here. Fixed the same way: generate the id client-side, never chain `.select()` after the insert. `learning-groups` and the rest of `announcements/actions.ts` were checked and found already safe.
3. **`PushSubscriptionControls.tsx` had a real React hydration mismatch on every page load**, found during the dedicated Web Push browser verification pass. `isAvailable` (browser feature + VAPID key detection) was computed via a synchronous `useMemo` calling `isPushSupported()` during render; that function reads `window`/`navigator`, so it returned `false` during SSR but `true` on the client's first render, mismatching the enable button's `disabled` attribute (visible as a repeatable React console error on every load). Fixed with `useSyncExternalStore` and an explicit `false` server snapshot — the standard pattern for values that legitimately differ between server and client. An initial `useEffect` + `setState` fix was rejected by the `react-hooks/set-state-in-effect` lint rule and replaced with this approach.
4. **`CalendarDateNavigator.tsx`'s date-range label rendered in reversed visual order under RTL**, found during the Admin Calendar Views v2 browser smoke test. The week/day/month label span had no explicit `dir`, so under the page's `dir="rtl"` context the browser's Unicode bidi algorithm displayed a range like `"5.7 - 11.7.2026"` (correct DOM text, confirmed via direct inspection) visually as `"1.7.2026 - 5.7"` (reversed). Fixed by adding `dir="ltr"` to that span, since date/time tokens are always LTR-formatted regardless of UI locale. Verified fixed live: the same week now renders in the correct visual order.
5. **The audit log's `toDate` filter silently excluded real same-day rows**, found during the Admin Audit Log Viewer v2 browser smoke test, in both `getAdminAuditLogs` (`admin-queries.ts`) and the CSV export route (`route.ts`). Both computed the exclusive next-day upper bound via `new Date(`${toDate}T00:00:00`)` — a string with no `Z`/offset, parsed by Node as **local server time**. On this server (`Asia/Jerusalem`, UTC+3), converting that local-time `Date` back to UTC via `.toISOString()` silently shifted the boundary earlier by the timezone offset, so filtering to "today" excluded rows created earlier that same day (confirmed live: `toDate=2026-07-09` showed 0 of 28 real rows, `toDate=2026-07-10` showed all 28). `fromDate` was unaffected, since it builds its UTC boundary directly as a string without any local-time `Date` parsing. Fixed in both files by computing the boundary entirely in UTC via `Date.UTC(year, month - 1, day + 1)`. Verified fixed live in both the page and a direct `fetch` against the export route.

All five fixes were verified live in the browser afterward. The latest Web Push v1 task added one migration and reran `supabase db reset`, type generation, `npm run check:no-hebrew-in-code`, `npm run lint`, `npm run build`, and `git diff --check`; all passed, with line-ending warnings only on `git diff --check`. The dedicated Web Push browser verification pass, the Admin Calendar Views v2 smoke test, the Learning Groups Timetable Views v2 smoke test, and the Admin Audit Log Viewer v2 smoke test each reran `npm run check:no-hebrew-in-code`, `npm run lint`, `npm run build`, and `git diff --check` after their fixes (or, for the learning-groups pass, with no fixes needed); all passed (no schema changes in any pass). ## Staff App Redesign v1 Feedback Fixes Handoff

Four follow-up passes after real usage of the deployed Staff App Redesign v1. Full report: `docs/30_STAFF_APP_REDESIGN_V1_FEEDBACK_FIXES.md`; parallel handoff: `docs/parallel/GPT_STAFF_APP_REDESIGN_V1_FEEDBACK_FIXES_HANDOFF.md`.

- **Round 1 — 8 items of direct feedback**: bottom-nav icons rewritten to match the mockup's exact SVG path data (previously approximated with lucide-react); Settings now shows a general "ניהול מערכת" admin entry point for any manager, not just super-admins (who were previously the *only* role with any UI path into `/admin/*`, despite managers having real RLS access there — confirmed this narrow gating predated the redesign, not a regression); push-notification configuration investigated and confirmed already fully implemented and working (no code change — if broken in production, most likely a missing `NEXT_PUBLIC_VAPID_PUBLIC_KEY` env var on the deploy platform, outside this repo's visibility); Settings logo replaced with "אפליקציית נעה - גרסה 0.1"; a school emblem finally added to every tab header (the mockup always specified this; it had never been wired up), surfacing and fixing a second broken-SVG-asset bug (empty fill style) of the same kind found in the two logo SVGs during the original redesign pass; "תלמיד/תלמידים" replaced with "חניך/חניכים" throughout `he.json` (confirmed the mockup's own source already used this wording — an inconsistency from the original pass, not just a fresh request); student card restructured into 3 tabs (overview/goals/update-details), consolidating what used to be a single long page with inline edit forms scattered throughout.
- **Round 2 — production session-loss root cause (diagnosis later corrected — see Round 3)**: Ronen reported the session-loss symptom from Round 1's feedback persisted more broadly after Round 1 shipped, proving Round 1's fix addressed only one instance of a systemic bug. This round claimed memoizing `createClient()` via React's `cache()` closed cross-request refresh races. **That claim was incorrect and was corrected in Round 3**: `cache()` only dedupes within one Server Component render pass / Server Action invocation / Route Handler invocation and cannot coordinate separate requests. The `AdminShell.tsx` `prefetch={false}` fix from this round (reducing a burst of ~7 simultaneous auth-middleware requests from sidebar link prefetching) was real and is retained.
- **Round 3 — corrected root cause, redone from the actual library source**: after a detailed technical correction from Ronen, redone by reading `src/proxy.ts` and the installed `@supabase/ssr@0.12.0`/`@supabase/supabase-js@2.110.0` types directly rather than by inference. Found and fixed four confirmed bugs: (1) `redirectWithCookies` in `src/proxy.ts` forwarded only cookie name+value on redirects, silently dropping `path`/`httpOnly`/`secure`/`sameSite`/`maxAge` on every redirect that followed a token refresh — fixed to forward the full cookie object; (2) the CDN no-cache headers (`Cache-Control: private, no-cache, no-store...`) that `@supabase/ssr` passes via a `setAll(cookiesToSet, headers)` second parameter — specifically to stop a CDN/edge layer from caching and replaying a response that just set fresh auth cookies — were silently dropped because `setAll` only declared one parameter; this app is on Vercel, which has exactly the CDN layer local dev doesn't, making this the leading candidate for why the symptom was production-only and correlated with longer sessions; (3) an RPC error from `current_user_is_active_staff` was indistinguishable from a genuine "not active staff" result, forcing an `/access-pending` redirect over a transient DB/network failure — fixed to pass the request through untouched instead (same fix applied to the equivalent path in `src/lib/auth/session.ts`, used by the root `/` page); (4) switched `getUser()` to `getClaims()`, which verifies locally via WebCrypto/JWKS (confirmed this project's tokens use asymmetric ES256 signing) instead of a network round-trip on every request. `cache()` in `src/lib/supabase/server.ts` was corrected to a within-invocation-only dedup optimization with an explicit comment that it provides no cross-request coordination — session correctness no longer depends on it in any way. `BottomNav.tsx`'s unread-count fetch (a separate Server Action firing on nearly every navigation) was deferred 400ms as an additional practical mitigation.
- **Local expiry test result, framed correctly**: per explicit instruction not to declare success from a fresh-token walkthrough alone, ran a real deterministic expiry test (`supabase/config.toml`'s `jwt_expiry` temporarily set to 15 seconds, restored afterward, confirmed via `git diff --stat` showing no residual change). A single authenticated request past-expiry correctly refreshes and succeeds. Multiple authenticated requests landing close together in time occasionally produced `refresh_token_already_used`. **This was an intentionally extreme stress test, not a realistic scenario**: Supabase recommends not running production expiry below five minutes, and at 15 seconds essentially every request — including the app's own incidental background ones — attempts a refresh, manufacturing far more collision opportunities than production's ~1-hour cycle would. A residual risk was observed only under this intentionally unrealistic 15-second stress test; its relevance to production is unknown, and this result does not by itself prove anything about production behavior. It does not, on its own, justify new coordination infrastructure — see the diagnostic path below.
- **Round 4 — pre-deploy closeout**: removed React `cache()` from `src/lib/supabase/server.ts` entirely (rather than retaining it as a "dedup optimization" as Round 3 did) — a generic factory shared by Server Components, Server Actions, and Route Handlers shouldn't depend on a React-render-scoped primitive at all, regardless of caveats in a comment. `createClient()` is now a plain uncached `async function`. The one real duplicate-client case, `getMessagesFeed()`, now explicitly creates one client and passes it into `getAnnouncements()`/`getNotifications()` (both take an optional shared-client parameter, still creating their own when called standalone elsewhere). The `[proxy-auth]` diagnostic logging is now gated behind a server-only `AUTH_DIAGNOSTICS_ENABLED` env var (unset = silent, documented in `.env.example`) instead of always-on. `BottomNav.tsx`'s deferred-fetch safety was re-verified line-by-line (timer cancellation, no post-unmount state updates, no firing on admin routes, no firing on unrelated navigations) and its comment reworded to state it's a mitigation, not a claim that it resolves refresh races.
- **Round 5 — production logout fixes**: resolved the production session-loss redirect bug. (1) Disabled prefetch on Notifications and Sign Out Links on Settings page (`prefetch={false}`). (2) Stopped BottomNav's initial-mount background unread count fetch (removed `prevPath === null` as an automatic reason to fetch) to prevent background auth requests on page loads. (3) Classified background requests (RSC/prefetch/Server Action/Accept non-HTML) in `src/proxy.ts` and allowed them to pass through safely without a redirect on claims failures, returning a controlled non-redirect response and logging as `background_claims_error`. (4) Reduced page-level Auth-server calls by replacing `getUser()` with cryptographic claims lookup `getClaims()` and using `claimsData?.claims?.sub` in profiles/notifications queries, admin-queries, and notification actions.
- **Verification**: `tsc --noEmit`, `npm run lint`, `npm run check:no-hebrew-in-code`, `npm run build` all pass clean after every change across all five rounds. Live-verified fresh-session walkthroughs with no regressions; anonymous users still redirect to `/login`; inactive profiles still land on `/access-pending`, not `/login`; a working screenshot confirmed the new icons/emblem render correctly.
- **Scope preserved**: no RLS, migrations, or server-action business logic changed — this stayed a visual/IA/auth-plumbing fix, no new features. Nothing committed by this session.
